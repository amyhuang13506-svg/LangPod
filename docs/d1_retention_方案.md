# D1 留存修复方案（2026-07-17）

> **⚠️ 重要修正（同日）**：本方案初版基于"新用户第一屏 = 学习播客"的假设写成。
> 实际上首页默认 segment 第一个区块是**原声播客（YouTube 源，OSS 播放）**，
> 且该链路此前零埋点 —— 新用户主路径在友盟里完全不可见。已补三个事件：
> `raw_play_start` / `raw_listen_30s` / `raw_listen_end`（含杀进程补报）。
> 由此新增一个候选诊断：**Onboarding 承诺"按级别定制的学习计划"，落地却是
> 母语级原声内容 —— 承诺与首屏体验断裂，且初学者面对最难内容直接流失。**
> B 层方案需等新漏斗数据确认：新用户到底在哪个 surface（原声 vs 学习播客）流失。

## 背景：数据在说什么

- 7-14 / 7-15 投放起量（118 / 113 新增），D1 留存 28.81% / 15.04%，7-14 批次 D2 断崖到 8.47%
- 平均单次使用 1分18秒 —— 新用户连一集的第 1 遍都没听完就走了
- 人均日使用时长 -34.78% 是新流量拉低均值，不是老用户流失

## 埋点审计结论（已完成，数据可信）

- 漏斗五事件（launch → onboarding → play_start → complete → first_complete）触发时机全部正确，无重复计数
- **但 episode_complete 门槛 = 免费用户连续听 4 遍（12-20 分钟）**，与 1分18秒的现实完全脱节：
  - 新用户几乎 0% 触发完播 → 生词卡奖励（aha）永远看不到
  - `first_episode_complete` 作为 Adjust/FB 优化信号基本不触发，投放算法学不到"什么样的用户是好用户"
- play_start → complete 之间无任何中间事件，无法定位用户具体在第几秒/第几遍流失

## 解决方案（三层，按优先级）

### A. 测量修复（先做，半天，不改产品行为）

改完立刻部署，让 7-18 起的数据能精确定位流失点。

1. **新增 `episode_round1_complete` 事件**
   - 挂在 AudioPlayer.advancePhase 的 `.englishRound(1)` 分支（已有 taskEventFirstRoundFinished 通知，旁边加一行 Analytics.track）
   - 参数：episode_id / level
2. **新增 `listen_30s` 事件**（每集每日一次）
   - 播放累计到 30 秒时触发。这是"用户给了产品一次机会"的最浅信号
3. **新增 `episode_abandon` 事件**
   - 切集/退后台/停止时，若未完播则上报：listened_seconds / phase（第几遍）/ level
   - 这是回答"用户死在哪一秒"的直接证据
4. 漏斗从 3 层变 6 层：
   `play_start → listen_30s → round1_complete → complete → first_complete`（+ abandon 分布）

### B. 首次会话改造（核心，1-2 天，需确认）

问题本质：**aha 时刻（听懂 + 生词卡）藏在 12-20 分钟后，新用户 1 分钟就走了。**

1. **新用户首集轻量模式**：设备上前 N 集（建议 N=1）播放流程改为
   `英语 ×1 → 中文 ×1 → 生词卡`（Easy 级约 6-10 分钟 → 缩到 3-5 分钟仍嫌长，见 2）
2. **【推荐】首集用 60-90 秒超短 demo**：
   - pipeline 单独生成一集 60-90 秒迷你集（3-4 句对话），走完整流程：英 ×1 → 中 ×1 → 3 张生词卡
   - onboarding 最后一页（plan 页之后）直接自动开播这集 —— 用户 2 分钟内体验完整闭环
   - 之后回到正常内容和正常遍数
3. **遍数进度可视 + 可跳过**：播放页"第 2/4 遍"旁加「跳到中文翻译」按钮（所有用户）
   - 重复 3 遍是刻意设计（磨耳朵），但对没建立信任的新用户是劝退器；给出口比让他退 app 好
4. 完播判定同步放宽：新用户轻量流程播完也算 episode_complete（保住 first_episode_complete 信号）

### C. 投放信号 + 召回（配合 B，半天）

1. **Adjust 质量信号迁移**：`first_episode_complete` 触发率起来之前，先把 `episode_round1_complete`（或 listen_30s）配上 Adjust token 作为过渡优化事件，让 FB/抖音的算法有东西可学
2. **D1 召回推送验证**：拉 7-14/15 批次的 push_opened 分 intent 数据，确认推送真的在发；若发送量正常但点击低，改新用户专属文案（"昨天那集 XX 还有 3 个生词没看"）

## 不改的东西（决策记录）

- 免费每日 2 集限额不动 —— 不是 D1 杀手（新用户连 1 集都没听完）
- Onboarding 6 页暂不砍 —— 先看漏斗里 launch → onboarding_complete 的真实转化率再决定
- 老内容不重新生成 —— demo 集单独生成一集即可

## 执行顺序

1. A 层今天做完 + 部署真机（不改任何产品行为，无风险）
2. B 层等确认方案细节（demo 集 or 轻量首集 or 两个都要）
3. 明天起每天看新漏斗数据，用数据决定 B 层是否加码
